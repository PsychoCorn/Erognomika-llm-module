
# LLM-модуль для agile-доски задач

## Назначение

Этот модуль отвечает за интеграцию LLM в API agile-доски задач.

Основные сценарии:
- обработка пользовательского сообщения в формате чата;
- генерация автоматического подбадривающего сообщения;
- возврат строго валидированного JSON в едином контракте.

Модуль спроектирован так, чтобы:
- не хардкодить схему ответа внутри prompt;
- использовать Pydantic-схемы как единый источник истины;
- изолировать работу с моделью от маршрутов FastAPI;
- упростить интеграцию в более крупное приложение.

## Кратко по файлам

### `schemas.py`

Главный файл с контрактами данных.

Здесь находятся:

* схемы задач;
* схемы контекста;
* схемы входных запросов;
* схема ответа LLM.

Это **ключевой файл контракта**.

Менять здесь можно:

* состав полей задач;
* состав входного контекста;
* состав ответа модели.

Важно:

* изменения в `schemas.py` автоматически влияют на prompt, потому что prompt берет JSON Schema динамически;
* если меняется схема, нужно проверять совместимость интеграции, клиентского кода и бизнес-логики.

Сюда **лезть можно и нужно**, если требуется менять контракт данных.

---

### `llm.py`

Внутренняя логика работы с моделью.

Здесь находится:

* сборка prompt для чата;
* сборка prompt для подбадривающего сообщения;
* вызов модели через OpenAI SDK;
* валидация ответа модели;
* сервисные функции высокого уровня для routes.

Сюда обычно **лезть не нужно**, если:

* меняются только схемы;
* меняются только route-обработчики;
* меняется только внешняя интеграция.

Сюда стоит лезть, если нужно:

* поменять prompt-логику;
* поменять стратегию вызова модели;
* сменить провайдера;
* добавить retry / logging / tracing / fallback;
* изменить обработку ошибок на уровне LLM.

---

### `routes.py`

HTTP-маршруты FastAPI для доступа к LLM-функциональности.

Здесь:

* принимаются входные запросы;
* вызываются сервисные методы из `llm.py`;
* ошибки преобразуются в HTTP-ответы.

Сюда можно лезть, если нужно:

* изменить URL-структуру;
* добавить новые эндпоинты;
* повесить авторизацию;
* добавить dependency;
* встроить модуль в более крупное API.

Сюда не стоит тащить:

* тяжелую prompt-логику;
* парсинг и валидацию ответа модели;
* прямой бизнес-пайплайн работы с LLM.

Идея такая: `routes.py` должен оставаться максимально тонким.

---

### `config.py`

Конфигурация модуля через environment variables и `pydantic-settings`.

Здесь задаются:

* `llm_api_key`
* `llm_model`
* `llm_base_url`
* `llm_temperature`
* `llm_max_tokens`
* `llm_timeout`

Сюда лезть нужно, если:

* добавляются новые настройки;
* меняются дефолты;
* нужно читать `.env`.

---

### `main.py`

Точка входа FastAPI-приложения.

Нужен не потому, что имя `main` магическое, а потому что где-то должен жить объект:

```python
app = FastAPI(...)
```

Если модуль подключается в большое приложение, `main.py` может быть вообще не нужен, а использоваться будет только `router` из `routes.py`.

Примеры:

* самостоятельный запуск: `uvicorn src.main:app --reload`
* встраивание в большое приложение: `app.include_router(llm_router)`

---

### `__init__.py`

Технический файл пакета.

Обычно:

* либо пустой;
* либо просто реэкспортирует `app` или `router`.

Логику приложения сюда лучше не складывать.

## Куда лезть, а куда нет

### Обычно менять можно

* `schemas.py` — если меняется контракт;
* `routes.py` — если меняется HTTP-интеграция;
* `config.py` — если меняются настройки;
* `main.py` — если модуль запускается как отдельное приложение.

### Обычно лучше не трогать без необходимости

* внутренности `llm.py`, особенно:

  * `_call_model`
  * `parse_llm_response`
  * сервисные методы генерации ответов

Потому что именно там находится критическая логика:

* сборка prompt;
* вызов модели;
* строгая валидация контракта.

Если сломать это место, модуль может:

* начать принимать невалидный JSON;
* перестать соблюдать схему;
* вернуть неподходящий язык ответа;
* сломать обработку задач.

## Как устроен контракт LLM

Модель должна возвращать **только JSON**.

Формат ответа задается схемой `LLMResponse` из `schemas.py`.

Допустимые ключи:

* `message_to_user`
* `add_tasks`
* `change_tasks`
* `delete_tasks`

### Семантика полей

#### `message_to_user`

Текстовый ответ пользователю в чат.

Ожидается, что:

* он написан на языке пользователя;
* он не содержит служебных пояснений;
* он является обычным пользовательским ответом.

#### `add_tasks`

Список новых задач на создание.

Каждый объект должен соответствовать схеме для добавления задачи.

#### `change_tasks`

Список существующих задач на изменение.

Каждый объект должен содержать `id` задачи, которую нужно изменить.

#### `delete_tasks`

Список `id` задач для удаления.

Важно:

* порядок имеет значение;
* сначала должны идти дочерние задачи, потом родительские.

---

## Почему схема не хардкодится в prompt

В `llm.py` prompt строится не из вручную написанного JSON-примера, а из:

```python
LLMResponse.model_json_schema()
```

Это значит:

* изменение `schemas.py` автоматически отражается в prompt;
* не нужно синхронно править схему в двух местах;
* меньше риск расхождения между реальной валидацией и инструкцией для модели.

Это один из ключевых архитектурных принципов модуля.

## Routes

Базовый префикс:

```text
/llm
```

### `GET /llm/health`

Простейший healthcheck.

#### Ответ

```json
{
  "status": "ok"
}
```

Используется для:

* проверки, что сервис поднялся;
* простого мониторинга;
* smoke test.

---

### `POST /llm/chat`

Основной endpoint для обработки пользовательского сообщения.

#### Назначение

Принимает:

* новое сообщение пользователя;
* контекст проекта;
* историю чата;
* список задач;
* текущую дату.

Передает это в LLM и возвращает строго валидированный `LLMResponse`.

#### Вход

Схема: `ChatRequest`

Внутри:

* `user_message`
* `context`

#### Выход

Схема: `LLMResponse`

#### Пример сценария

Пользователь пишет:

> Разбей задачу по интеграции DeepSeek на 3 подзадачи

Модуль может вернуть:

* текстовый ответ в `message_to_user`;
* набор задач в `add_tasks`.

---

### `POST /llm/encouragement`

Endpoint для генерации автоматического подбадривающего сообщения.

#### Назначение

Используется внешней логикой, когда нужно автоматически сгенерировать поддерживающее сообщение пользователю.

Важно:

* сам модуль не решает, **когда** именно вызывать этот endpoint;
* он только генерирует ответ, если его вызвали.

#### Вход

Схема: `EncouragementRequest`

#### Выход

Схема: `LLMResponse`

В типичном случае:

* заполнено только `message_to_user`;
* списки действий пустые.

## Как идет поток данных

### Сценарий `/llm/chat`

1. Клиент или внешний backend вызывает route `/llm/chat`
2. FastAPI валидирует тело запроса через `ChatRequest`
3. `routes.py` вызывает `generate_chat_response(...)`
4. `llm.py` строит prompt через `build_prompt(...)`
5. `llm.py` вызывает модель через `_call_model(...)`
6. raw-ответ модели парсится
7. ответ валидируется через `LLMResponse`
8. валидированный JSON возвращается клиенту

### Сценарий `/llm/encouragement`

Поток тот же, только используется:

* `EncouragementRequest`
* `build_encouragement_prompt(...)`
* `generate_encouragement_response(...)`

## Как подключить модуль в большое приложение

Если у вас уже есть основной FastAPI-сервис, этот модуль можно не запускать отдельно.

Достаточно подключить router.

Пример:

```python
from fastapi import FastAPI
from src.routes import router as llm_router

app = FastAPI()
app.include_router(llm_router)
```

Тогда routes модуля станут частью общего API.

Если модуль нужен как отдельный сервис:

uvicorn src.main:app --reload

## Что обязательно проверить после интеграции

### 1. Валидируется ли входной запрос

Проверить, что `ChatRequest` и `EncouragementRequest` реально соответствуют тому, что отправляет внешний код.

### 2. Возвращает ли модель JSON без мусора

Некоторые модели любят:

* оборачивать JSON в markdown;
* писать пояснения до/после;
* добавлять лишние поля.

Модуль это режет валидацией, но лучше убедиться тестами.

### 3. Совпадает ли схема задач с реальной БД

Особенно важно проверить:

* `parent_id`
* `status`
* формат дат
* использование `project_id`

### 4. Правильно ли обрабатывается `delete_tasks`

Если в системе есть иерархия задач, удаление должно учитывать порядок:

* дети
* потом родители

### 5. Правильно ли выбран язык ответа

Промпт требует отвечать на языке пользователя, но это стоит дополнительно проверить на:

* русском;
* английском;
* смешанных сообщениях.

## Ограничения текущей реализации

На данный момент модуль:

* не решает сам, когда слать encouragement;
* не хранит историю в БД;
* не применяет изменения к задачам сам по себе;
* не содержит retry/policy/fallback;
* не делает аудит логов;
* не содержит auth/per-user isolation;
* не умеет стриминг ответа;
* не умеет частичное обновление задач как patch.

То есть модуль отвечает именно за:

* prompt;
* вызов модели;
* валидацию JSON-контракта;
* HTTP-обертку.

---

## Где дорабатывать дальше

### Если нужен production-уровень, то обычно добавляют

* structured logging;
* correlation id / request id;
* retry policy;
* timeout policy;
* rate limiting;
* auth;
* tracing;
* metrics;
* unit/integration tests;
* fallback-модель;
* защиту от повторной отправки одинаковых запросов;
* пост-валидацию бизнес-правил.

---

## Что важно помнить при изменении схем

Если меняется `schemas.py`, нужно понимать, что это влияет сразу на:

* валидацию FastAPI;
* JSON Schema, которая идет в prompt;
* итоговый контракт ответа модели;
* интеграцию с внешними обработчиками.

Поэтому любое изменение схемы — это фактически изменение API-контракта.

## Итог

Для интегратора главное правило такое:

* **контракт данных живет в `schemas.py`**
* **HTTP-вход живет в `routes.py`**
* **внутренняя LLM-логика живет в `llm.py`**
* **настройки живут в `config.py`**
* **`main.py` нужен только как точка входа, если модуль запускается отдельно**

Если модуль встраивается в большое приложение, чаще всего достаточно:

* импортировать `router`
* подключить его через `include_router(...)`
* настроить `.env`
* убедиться, что внешний код шлет данные в формате схем
